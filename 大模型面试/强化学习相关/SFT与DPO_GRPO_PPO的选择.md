# 什么时候使用 SFT，什么时候使用 DPO/GRPO/PPO？

在大型语言模型（LLM）的训练和微调过程中，**SFT（有监督微调）** 和 **对齐/强化学习阶段（DPO/GRPO/PPO）** 扮演着完全不同但互补的角色。

简单来说：**SFT 是“教模型学会怎么说话和遵循指令”，而 DPO/GRPO/PPO 是“教模型说符合人类偏好的好话，或者在复杂任务中探索出最优解”。**

以下是关于什么时候使用哪种方法的详细分析和总结：

---

### 一、 什么时候使用 SFT (Supervised Fine-Tuning)？

SFT 是指令微调的基础，本质上是**行为克隆（Behavior Cloning）**。你给模型提供高质量的“输入-输出”对，模型通过交叉熵损失函数学习模仿这些回答。

**适用场景：**
1. **基础指令遵循（Instruction Following）：** 让预训练模型（Base Model）学会问答格式，理解什么是“指令”，什么是“回答”。
2. **领域知识注入与格式定制：** 当你需要模型输出特定格式（如 JSON、特定的 Markdown 结构），或者适应特定垂直领域（如医疗问答、法律文书生成）的行文风格时。
3. **冷启动阶段：** 在进行任何强化学习（RLHF）之前，必须先有一个质量过关的 SFT 模型作为起点（Actor 模型的初始化）。
4. **资源有限且任务明确：** SFT 训练相对稳定，不需要复杂的奖励模型（Reward Model），超参数好调。如果你的任务有明确的“标准答案”，且容易收集高质量数据，SFT 通常就足够了。

**局限性：** SFT 只能让模型“模仿”数据，无法让模型超越演示数据的上限；且面对开放性问题时，SFT 无法有效惩罚模型的“坏行为”（如幻觉、有害输出）。

---

### 二、 什么时候使用 DPO / GRPO / PPO？

这些算法属于**人类偏好对齐（Alignment）**或**强化学习（RL）**阶段。当 SFT 已经让模型“会说话”后，我们需要用这些方法来优化模型的内在逻辑、价值观或推理能力。

#### 1. 什么时候使用 DPO (Direct Preference Optimization)?
DPO 是一种**离线偏好优化**方法。它巧妙地绕过了训练奖励模型（Reward Model）的步骤，直接使用“好回答（Chosen）”和“坏回答（Rejected）”的数据对来更新模型。

* **适用场景：**
  * **价值观与偏好对齐：** 调整模型的语气（如更礼貌）、安全性（拒绝回答有害问题）、减少幻觉。
  * **已有高质量的偏好数据集：** 你手头有大量的 `(Prompt, Chosen, Rejected)` 数据。
  * **算力/显存受限：** 相比 PPO，DPO 不需要同时加载 Reward Model 和 Value Model，显存占用小得多，训练也更稳定（不需要调复杂的 RL 超参数）。
* **局限性：** 它是离线算法，模型只能在给定的偏好数据对中学习，缺乏主动探索（Exploration）新答案的能力，对于提升复杂的逻辑推理能力帮助有限。

#### 2. 什么时候使用 PPO (Proximal Policy Optimization)?
PPO 是经典的 RLHF 算法（如 ChatGPT 早期使用的方案）。它需要四个模型参与：Actor（策略模型）、Reference（参考模型）、Reward（奖励模型）和 Critic/Value（价值模型）。

* **适用场景：**
  * **开放式生成与复杂对齐：** 当你需要模型在庞大的解空间中进行探索，并且你有一个非常强大的 Reward Model 可以给出准确打分时。
  * **追求极致性能上限：** PPO 允许模型在训练过程中不断生成新回答并获取反馈，这种“在线探索”能力使其在很多场景下上限高于 DPO。
* **局限性：** **极其昂贵且难以训练**。同时加载 4 个模型对显存要求极高；强化学习的超参数（如 KL 惩罚系数、学习率、优势函数计算）非常敏感，容易出现“奖励骇客（Reward Hacking）”现象（即模型找到了作弊拿高分的方法，但输出变成了乱码）。

#### 3. 什么时候使用 GRPO (Group Relative Policy Optimization)?
GRPO 是 DeepSeek（在 DeepSeekMath 和 DeepSeek-R1 中）提出并带火的算法。它针对 PPO 进行了改良，**去掉了庞大的 Critic/Value 模型**，改为对同一个 Prompt 生成一组（Group）回答，然后在组内计算相对优势（Advantage）。

* **适用场景：**
  * **复杂推理任务（数学、代码、逻辑）：** 这类任务有明确的规则（Rule-based Reward），比如代码是否能编译通过、数学题最终答案是否正确。GRPO 非常适合用来激发模型的“顿悟（Aha-moment）”和长链条推理（CoT）能力。
  * **想要 PPO 的探索能力，但显存不够：** GRPO 省去了与 Actor 同等规模的 Value Model，大大降低了显存开销，使得在有限算力下进行大规模 RL 成为可能。
* **局限性：** 依赖于组内采样（通常一个 Prompt 采样 4-8 个回答），如果任务本身没有客观的评判标准（比如写诗、闲聊），GRPO 的组内相对评分可能不如 PPO + 强大 Reward Model 稳定。

---

### 三、 总结与决策树

在实际的 LLM 开发中，它们通常是**串联**使用的，而不是非此即彼。标准的现代大模型训练范式是：**Base Model -> SFT -> (DPO 或 RLHF/GRPO)**。

**决策指南：**

1. **第一步：你有没有高质量的问答数据？**
   👉 **必须先做 SFT。** 无论后续用什么，SFT 都是基础。

2. **第二步：SFT 做完后，模型表现不错，但有时会产生幻觉、语气不对或不够安全？**
   👉 **使用 DPO。** 收集一些“好 vs 坏”的对比数据，跑一下 DPO，成本低、见效快、训练稳定。目前开源社区 80% 的对齐工作都是用 DPO（或其变体如 ORPO/KTO）完成的。

3. **第三步：你的目标是提升模型的“硬核推理能力”（如解数学题、写复杂代码），且有自动化的判题系统（规则奖励）？**
   👉 **使用 GRPO。** 它可以让模型通过不断试错（RL 探索）学会自我反思和长思考（类似 DeepSeek-R1 的 `<think>` 过程），且比 PPO 节省显存。

4. **第四步：你有充足的算力、顶级的算法团队，且需要对齐极其复杂的开放域人类偏好？**
   👉 **使用 PPO。** 训练一个强大的 Reward Model，然后用 PPO 榨干模型的最后一点性能潜力。
